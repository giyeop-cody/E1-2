# JSON 파일 스키마


 1. 퀴즈 콘텐츠 (quizzes)

    실제 퀴즈 문제들이 담기는 배열(Array) 형태의 리스트입니다.

    id: 각 문제를 식별하는 고유 번호입니다.

    question: 사용자에게 보여줄 질문 텍스트입니다.

    choices: 객관식 선택지 리스트입니다.

    answer: 정답의 인덱스입니다. (예시에서는 1이므로 첫 번째 항목인 "Guido"가 정답임을 의미합니다.)

    hint: 오답을 골랐거나 요청 시 제공할 힌트 메시지입니다.

2. 최고 기록 (best_score)

    사용자 혹은 시스템 전체에서 달성한 가장 높은 점수를 저장하는 숫자 데이터입니다.

3. 게임 이력 (records)

    과거에 퀴즈를 풀었던 세부 정보를 담은 객체 리스트입니다.

    timestamp: 퀴즈를 수행한 날짜와 시간입니다.

    score: 해당 회차에서 획득한 점수입니다.

    solved: 정답을 맞힌 개수입니다.

    tried: 전체 도전한 문제 수입니다.

```
{
    "quizzes": [
        {
            "id": 1,
            "question": "Python의 창시자는?",
            "choices": ["Guido", "Linus", "Bjarne", "James"],
            "answer": 1,
            "hint": "네덜란드 출신의 프로그래머입니다."
        }
    ],
    "best_score": 3,
    "records": [
        {
            "timestamp": "2026-04-10 14:00:00",
            "score": 470,
            "solved": 5,
            "tried": 10
        }
    ]
}
```

# 예기치 못한 상황에 crash 없이 안전 종료 처리 (Graceful Shutdown)

* 사용자가 `Ctrl+C`(`SIGINT`) 등의 키보드 인터럽트를 입력했을 때 또는 표준 입력의 `eof`를 받았을 때, 프로그램이 갑자기 비정상적으로 꺼지는 것이 아니라 하던 일을 마무리하고 안전하게 종료되도록 로직을 추가해야 합니다.


* 핵심

    예외 처리 (Exception Handling): Ctrl+C는 특정 예외(Python의 KeyboardInterrupt)를 발생시킵니다. 이를 try-catch 블록으로 잡아내야 합니다.

* 자원 해제: 종료 직전에 열려 있는 파일(json 파일)을 닫는 코드가 포함되어야 합니다.

* 사용자 메시지: "프로그램을 안전하게 종료합니다..."와 같은 메시지를 출력하여 사용자에게 상태를 알립니다.

# 커밋 단위 및 메시지 규칙 (Commit Convention)
* 커밋 단위 : 하나의 기능 구현 또는 하나의 버그 수정
* 커밋 메시지 구조 : type: subject 형식

| Type | 설명 |
| :--- | :--- |
| **feat** | 새로운 기능 추가 |
| **fix** | 버그 수정 |
| **docs** | 문서 수정 (README, 위키 등) |
| **style** | 코드 포맷 변경, 세미콜론 누락 등 (코드 로직 변경 없음) |
| **refactor** | 코드 리팩토링 (기능 변경 없이 코드 구조 개선) |
| **test** | 테스트 코드 추가 및 수정 |
| **chore** | 빌드 업무 수정, 패키지 매니저 설정, 환경 설정 변경 등 |

# 만약 1000개 이상의 퀴즈를 넣을 경우 JSON의 한계와 해결법

## 한계
* I/O 오버헤드 증가
    전체 JSON 파일을 메모리에 로드 후 다시 통째로 저장하는 구조로 입출력 시간이 길어지고 메모리 점유율이 증가함
* 검색 및 필터링 속도 저하
    검색 및 삭제에서 O(n)의 시간 복잡도로 전체를 순회하기에 시간이 길어짐

## 해결
* 파일을 분할하여 저장하여 부분만 읽어서 로딩하고 메모리에 저장하는 방식을 사용
    Quiz1.json, Quiz1.json, Ranking.json
* JSON 파일 대신에 DB를 사용

|구분|JSON (현재 방식)|DB (대안)|
| :--- | :--- | :--- |
|적정 규모|수십 ~ 수백 개|수만 개 이상 가능|
|검색 속도|느림 (전체 순회 필요)|매우 빠름 (인덱싱 활용)|
|안정성|파일 깨짐 위험 높음 (쓰기 중 중단 시)|트랜잭션 지원으로 매우 안전함|
|난이도|매우 쉬움 (텍스트 기반 관리)|중간 (SQL 문법 학습 필요)|

# 만약 state.json 손상될 경우 어떻게 복구할 것인가?
QuizGame 클래스(엔진)이 초기화되면 코드에서 해당 파일의 존재 여부/권한/스키마를 검증하고 이상이 있으면 파일을 새로 만들고 그 안에 초기 퀴즈 5개와 최고 점수 0점, 빈 기록을 추가하는 로직을 넣어서 손상 시 초기 대이터로 돌아 갈 수 있도록 로직 추가

# 요구사항 변경 시 수정 범위 파악
### 퀴즈 구조 변경 시 어떤 클래스나 파일(모델/엔진 등)을 수정해야 하는지에 대한 아키텍처적 대응 방안

* 새로운 데이터 항목이 필요할 때 -> 모델(Quiz 클래스) 수정
      
    ex) 힌트 추가 Quiz 클래스에 hint 추가

* 사용자 인터페이스 개선 -> main.py 수정

    ex) 메뉴 구성이나 출력 형식을 수정

* 비즈니스 로직이 변한 경우 -> 엔진(QuizGame 클래스) 수정 

    ex) 게임 규칙의 변경, 점수 계산 방식 변경, hint 추가 등

* 저장소를 변경하는 경우 ->  json_manager.py 수정

    json 파일의 대용량 데이터 처리 한계로 더 이상 json을 사용하지 않고 DB를 사용한다고 가정할 경우 json_manager.py를 DB_manager.py로 바꾸고 DB에 관련된 함수로 수정해야함.
